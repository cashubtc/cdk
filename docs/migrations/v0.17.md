# Migration Guide: cdk-mintd to v0.17.0

## Summary

`cdk-mintd` now stores its authoritative configuration document in the primary
mint database. A normal daemon start no longer reads `config.toml` or applies
operational `CDK_MINTD_*` environment-variable overrides.

Existing mints must import their configuration once before starting v0.17:

```bash
cdk-mintd config validate --file /path/to/config.toml
cdk-mintd config init --file /path/to/config.toml
cdk-mintd
```

Do not run `config init` without first completing the preparation below. The
first v0.17 start applies the imported mint metadata and quote TTL to the
existing database.

## Before Upgrading

1. Record the current mint binary version.
2. Save the active configuration file and the environment variables used by
   the mint service.
3. If management RPC is enabled, record the current RPC-managed values before
   stopping the old daemon:

   ```bash
   cdk-mint-cli get-info
   cdk-mint-cli get-quote-ttl
   ```

   Include the relevant RPC address and TLS arguments used by your deployment.
   Copy the reported name, descriptions, MOTD, icon, URLs, contacts, terms of
   service URL, and quote TTL into the import document. The first database-backed
   start treats that document as authoritative.
4. Stop `cdk-mintd`.
5. Back up the complete mint database and working directory. For PostgreSQL,
   take a database backup using the normal PostgreSQL tooling. For SQLite, copy
   the database only after the daemon has stopped.

Keep this backup until the upgraded mint has been verified. Rolling back may
require restoring it because the new binary can run database migrations while
opening the database.

## Prepare the Import Document

Start from the configuration that was active immediately before shutdown.

### Move Operational Environment Overrides into TOML

Normal v0.17 startup does not apply operational environment-variable overrides.
Values such as the listen address, payment backends, limits, cache settings,
mint metadata, and quote TTL must be present in the import document.

Only bootstrap variables and variables explicitly referenced by `env:` remain
runtime inputs.

### Replace Literal Secrets with References

Secret fields must use one of these forms:

```toml
[info]
mnemonic = "file:/run/secrets/mint-mnemonic"

[database.postgres]
url = "env:CDK_MINTD_POSTGRES_URL"

[lnbits]
admin_api_key = "env:CDK_MINTD_LNBITS_ADMIN_API_KEY"
invoice_api_key = "file:/run/secrets/lnbits-invoice-key"
```

The rule applies to:

- mint seeds and mnemonics;
- primary and authentication PostgreSQL URLs;
- LNbits API keys;
- BDK and LDK RPC passwords and mnemonics; and
- Redis connection strings and cluster nodes.

Opaque environment and file secrets are read exactly, including leading or
trailing whitespace. Ensure secret files do not contain an unintended final
newline. Mnemonic fields are the exception and ignore surrounding whitespace.

Resolved secret values are never stored in or exported from the configuration
record.

### Preserve the Signing Identity

Use the same mint seed, mnemonic, or remote signatory that the existing mint
used. During initialization, `cdk-mintd` compares the configured signer's root
public key with the mint public key already stored in the database.

Choose only one signing mode. An embedded signatory requires `[info].seed` or
`[info].mnemonic` on the mint host. A remote signatory requires
`[signatory].enabled = true` and no local seed or mnemonic; its private keys
remain on the signatory host.

Initialization fails rather than attaching a different signer to an existing
mint. Signer migration is not supported by `config init` or `config apply`.

### Match the Primary Database Bootstrap

Configuration commands and normal startup must locate the existing primary
database before they can read its stored document.

For SQLite, use the existing working directory:

```bash
cdk-mintd --work-dir /var/lib/cdk-mintd config init --file /path/to/config.toml
```

For PostgreSQL, set the same bootstrap settings used by the document:

```bash
export CDK_MINTD_DATABASE=postgres
export CDK_MINTD_POSTGRES_URL='postgresql://...'
export CDK_MINTD_POSTGRES_TLS_MODE=...
export CDK_MINTD_POSTGRES_MAX_CONNECTIONS=...
export CDK_MINTD_POSTGRES_CONNECTION_TIMEOUT_SECONDS=...
```

The URL can be referenced from the TOML as
`env:CDK_MINTD_POSTGRES_URL`. PostgreSQL pool and TLS settings in the document
must match the bootstrap values used by the command.

Encrypted SQLite commands also require the existing
`--password <sqlcipher-password>` argument.

## Initialize and Start

Run local document validation first:

```bash
cdk-mintd config validate --file /path/to/config.toml
```

This resolves secret references and validates the signer and configured
backends without writing the primary database. A configured remote signatory
must be reachable.

Initialize the configuration record against the existing database:

```bash
cdk-mintd --work-dir /var/lib/cdk-mintd \
  config init --file /path/to/config.toml
```

`config init` refuses to replace an existing configuration record. If it reports
that configuration is already initialized, inspect it with `config show` rather
than retrying initialization.

Inspect the stored unresolved document:

```bash
cdk-mintd --work-dir /var/lib/cdk-mintd config show
```

Start the daemon with the same working directory, database bootstrap settings,
SQLCipher password, and secret-reference environment variables:

```bash
cdk-mintd --work-dir /var/lib/cdk-mintd
```

The first successful start applies the imported mint metadata and quote TTL and
marks the document applied. Later starts preserve RPC-managed canonical values
when management RPC is enabled.

## Verify the Upgrade

Before reopening the mint to users, verify:

- the mint URL and HTTP listen address;
- mint name, descriptions, MOTD, icon, URLs, contacts, and terms of service;
- quote TTL values;
- active keysets and the mint public key;
- every configured Lightning and on-chain payment backend;
- authentication, caching, metrics, and management RPC; and
- logs for configuration, signer, database, or backend errors.

Use `config show` to confirm that the intended document is active. Use
`config export --file <path>` to make a portable copy containing secret
references, not resolved values.

## Later Configuration Changes

Edit an import document, then explicitly stage it:

```bash
cdk-mintd config validate --file /path/to/changed-config.toml
cdk-mintd config apply --file /path/to/changed-config.toml
cdk-mintd config show
```

Restart the daemon to activate the replacement. Do not make management RPC
changes between `config apply` and that restart.

## Rollback

1. Stop the v0.17 daemon.
2. Restore the pre-upgrade database and working-directory backup.
3. Restore the previous configuration file, environment variables, and binary.
4. Start the previous version and verify its mint public key, keysets, mint
   metadata, quote TTL, and payment backends before reopening traffic.

Do not run the previous binary against a database modified by v0.17 unless that
version's database compatibility has been independently verified.
